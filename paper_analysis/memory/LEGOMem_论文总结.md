## 一、研究背景与动机

### 1.1 多Agent LLM系统的现状

LLM越来越多地被部署为Agent来自动化复杂的多步骤工作流（文档编辑、邮件处理、日历调度等）。当前系统通常采用**多Agent设计**：

- **中心编排器（Orchestrator）**：执行规划和子任务委派
- **专业化任务Agent（Task Agents）**：执行具体子任务（Word、Excel、Calendar、Email等）

### 1.2 核心问题：无状态性

> 当前多Agent系统**大部分是无状态的和事务性的**：每个任务从头解决，不复用先前经验。

这种**程序性记忆的缺失**限制了：
- 从过去经验中学习的能力
- 随时间积累执行技能的能力
- 对复杂工作流的持续改进

### 1.3 现有方法的不足

| 方法 | 局限 |
|------|------|
| Synapse, AWM | 针对**单Agent**系统设计，不解决多Agent的独特协调和专业化挑战 |
| A-MEM, Mem0 | 聚焦对话历史的**情景/语义记忆**，不是为Agent学习和工作流自动化设计 |
| 短期上下文优化方法 | 仅关注短期上下文，不涉及长期程序性记忆 |

**关键空白**：缺少为**多Agent系统**设计的**模块化、角色感知的程序性记忆**框架。

## 二、核心贡献：LEGOMem框架

### 2.1 设计理念

> **LEGOMem**是一个模块化程序性记忆框架，将过去的任务轨迹分解为**可复用的记忆单元**，并灵活地在编排器和任务Agent之间分配，以支持规划和执行。

**名称寓意**：像乐高积木一样，将过去经验**模块化重组**，灵活拼接以支持新任务。

### 2.2 核心特点

| 特点 | 描述 |
|------|------|
| **模块化记忆** | 将轨迹分解为全任务记忆和子任务记忆 |
| **角色感知分配** | 编排器接收全任务记忆，任务Agent接收子任务记忆 |
| **RAG层设计** | 作为现有多Agent系统之上的检索增强层 |
| **三种检索变体** | Vanilla、Dynamic、QueryRewrite |

## 三、问题建模

### 3.1 多Agent工作流自动化系统

基于Magentic-One架构，系统包含：
- **编排器** $A_{orch}$：生成高层计划，动态选择下一个Agent和子任务
- **任务Agent集合** $A = \{A_1, ..., A_k\}$：Word、Excel、Calendar、Email、System、OCR-PDF
- **外部环境** $\mathcal{E}$：Docker模拟应用环境

### 3.2 编排循环

在每个编排步骤 $t$：
1. 编排器提议下一个子任务：$s_t = \pi_{orch}(\sigma_t)$
2. 子任务分配给适当的任务Agent $A_j$
3. 任务Agent执行子任务，返回观察 $o_t$ 和执行摘要 $r_t$
4. 编排器更新状态：$\sigma_{t+1} = f(\sigma_t, r_t)$
5. 若进展停滞 → 重新规划 $\pi'$

### 3.3 多Agent程序性记忆的形式化定义

> **记忆存储** $M$ 是一组从过去执行中提取的**模块化记忆单元**的集合，包括：
> - **全任务记忆（Full memories）**：捕获编排计划和汇总的执行轨迹
> - **子任务记忆（Subtask memories）**：捕获Agent特定的子任务执行

## 四、LEGOMem框架详解

### 4.1 记忆构建（离线阶段）

从成功完成的任务轨迹中蒸馏结构化可复用的记忆单元：

```
成功执行轨迹
     ↓
  LLM转化
     ↓
┌──────────────────────────────────────┐
│          LEGOMem 记忆单元             │
│                                      │
│  high_level_plan: "1. 检查日历...     │
│                    2. 创建Excel..."   │
│                                      │
│  subtasks:                           │
│    - agent: calendar_agent           │
│      description: "提取会议详情"      │
│      steps: <think>...</think><action>...   │
│      observations: "成功列出事件"     │
│                                      │
│    - agent: excel_agent              │
│      description: "创建Excel文件"     │
│      steps: <think>...</think><action>...   │
│      observations: "成功创建文件"     │
│                                      │
│  final_answer: "会议议程已转换..."    │
│  reflections: "任务成功完成..."       │
└──────────────────────────────────────┘
```

**两种记忆类型**：

| 类型 | 内容 | 接收者 | 索引方式 |
|------|------|--------|---------|
| **全任务记忆** | 任务描述 + 高层计划 + 最终答案 + 反思 | 编排器 | $\phi(d)$：任务描述嵌入 |
| **子任务记忆** | 子任务描述 + Agent行为 + 工具使用 + 观察 | 任务Agent | $\phi(d_{subtask})$：子任务描述嵌入 |

**记忆库实现**：向量数据库（FAISS），使用text-embedding-3-large生成密集嵌入。

### 4.2 记忆增强推理（在线阶段）

```
新任务到来
     ↓
① 检索Top-K相关全任务记忆（基于任务描述嵌入相似度）
     ↓
② 分配全任务记忆给编排器 → 支持初始规划和动态编排
     ↓
③ 从全任务记忆中提取子任务记忆
     ↓
④ 分配子任务记忆给对应任务Agent → 支持执行级指导
     ↓
⑤ 编排器动态选择Agent、生成子任务
     ↓
⑥ 任务Agent使用子任务记忆执行 → 返回观察和摘要
     ↓
⑦ 编排器更新状态 → 继续或重新规划
```

## 五、三种LEGOMem变体

### 5.1 Vanilla LEGOMem

- **全局记忆库** $M$，使用任务描述嵌入索引
- 推理时检索全任务记忆 → 分配给编排器
- 子任务记忆从检索到的全任务记忆中**直接提取**并**静态分配**给相关Agent

**优点**：简单高效
**缺点**：当检索到的全任务记忆的子任务结构与当前任务不同时，可能无法提供相关的子任务记忆

### 5.2 LEGOMem-Dynamic

- 编排器记忆与Vanilla相同
- 维护**按Agent分段的子任务记忆库** $\{M_{A_j} | A_j \in A\}$
- 在**执行过程中动态检索**：编排器生成子任务 $s_t$ 后，计算其嵌入 $\phi(s_t)$，查询对应Agent的记忆库

**优点**：更精确的执行级指导，减少不相关记忆的噪声
**缺点**：每个编排步骤都需要重复的子任务嵌入和检索

### 5.3 LEGOMem-QueryRewrite

- 编排器记忆与Vanilla相同
- 在**规划阶段**使用查询重写LLM $\psi$，基于检索到的全任务记忆生成**子任务草案计划**
- 每个子任务 $s'_j$ 被嵌入并用于**在任务执行前**检索相关子任务记忆

**优点**：保持细粒度检索优势，避免运行时重复查询
**缺点**：需要额外的查询重写LLM调用

### 5.4 三种变体对比

| 变体 | 子任务检索时机 | 检索粒度 | 额外开销 |
|------|-------------|---------|---------|
| Vanilla | 推理开始时（一次性） | 粗粒度（静态分配） | 最低 |
| Dynamic | 执行过程中（每步） | 细粒度（即时匹配） | 较高 |
| QueryRewrite | 规划阶段（一次性） | 细粒度（重写匹配） | 中等 |

## 六、实验设置

| 维度 | 详情 |
|------|------|
| **数据集** | OfficeBench（300个办公自动化任务，训练148/测试152） |
| **难度等级** | Level 1（单应用）、Level 2（双应用）、Level 3（多应用工作流） |

| **基线方法** | No Memory、Synapse（原始轨迹）、AWM（提取的工作流） |
| **记忆规模** | 93个全任务记忆 + 250个子任务记忆 |

## 七、主要实验结果

### 7.1 总体性能对比

#### LLM团队（GPT-4o）

| 方法 | Level 1 | Level 2 | Level 3 | 总体 |
|------|---------|---------|---------|------|
| No Memory | 49.31 | 58.52 | 33.33 | 45.83 |
| Synapse | 59.72 | 75.56 | 43.50 | 58.11 |
| AWM | 54.17 | 58.52 | 35.03 | 48.03 |
| **LEGOMem** | **57.99** | **73.33** | **47.46** | **58.44** |
| LEGOMem-Dynamic | 56.25 | 75.56 | 43.79 | 57.12 |
| LEGOMem-QueryRewrite | 54.17 | 72.59 | 42.94 | 55.26 |

#### Hybrid团队（GPT-4o编排 + GPT-4o-mini执行）

| 方法 | Level 1 | Level 2 | Level 3 | 总体 |
|------|---------|---------|---------|------|
| No Memory | 45.14 | 48.89 | 16.95 | 35.31 |
| Synapse | 46.53 | 68.15 | 29.94 | 46.49 |
| AWM | 43.75 | 55.56 | 18.64 | 37.50 |
| LEGOMem | 49.31 | 62.22 | 36.16 | 48.03 |
| LEGOMem-Dynamic | 44.44 | 65.93 | 36.16 | 47.59 |
| **LEGOMem-QueryRewrite** | **47.22** | **66.67** | **40.11** | **50.22** |

#### SLM团队（GPT-4o-mini）

| 方法 | Level 1 | Level 2 | Level 3 | 总体 |
|------|---------|---------|---------|------|
| No Memory | 36.81 | 34.81 | 7.34 | 24.78 |
| Synapse | 36.81 | 42.22 | 20.90 | 32.24 |
| AWM | 35.42 | 36.30 | 12.99 | 26.97 |
| **LEGOMem** | **38.89** | **54.07** | **25.42** | **38.16** |
| LEGOMem-Dynamic | 38.89 | 50.37 | 27.12 | 37.72 |
| LEGOMem-QueryRewrite | 36.81 | 48.89 | 26.55 | 36.40 |

### 7.2 关键发现

1. **LEGOMem在所有场景一致优于基线**：
   - LLM团队：总体+12.61%绝对提升
   - Hybrid团队：+12.72%
   - SLM团队：**+13.38%**（最大提升）

2. **LEGOMem使小模型缩小与大模型的差距**：
   - Hybrid+QueryRewrite (50.22%) > 无记忆LLM团队 (45.83%)
   - SLM+Vanilla LEGOMem (38.16%) > 无记忆Hybrid团队 (35.31%)

3. **三种变体性能相近**：展示框架的鲁棒性

4. **Synapse在LLM团队有竞争力**（强模型可解释原始轨迹），但在Hybrid和SLM团队效果不稳定

### 7.3 案例分析

**任务**：查找最早邮件的日期

| 条件 | 行为 | 结果 |
|------|------|------|
| **无记忆** | 仅读取第一封邮件后停止（不完整规划） | ❌ 错误答案 |
| **有LEGOMem** | 系统性读取并比较所有邮件时间戳 | ✅ 正确答案 |

> LEGOMem改善推理一致性和任务完整性。

## 八、消融实验

### 8.1 记忆放置策略

| 记忆放置 | LLM团队总体 | Hybrid团队总体 |
|---------|-----------|--------------|
| **编排器+Agent记忆（完整）** | **58.44** | **48.03** |
| 仅编排器记忆 | 53.29 | 47.59 |
| 仅Agent记忆（Vanilla） | 49.78 | 35.31 |
| 仅Agent记忆（Dynamic） | 49.34 | 40.35 |
| 仅Agent记忆（QueryRewrite） | 50.66 | 39.69 |
| 无记忆 | 45.83 | 35.31 |

**关键发现**：

1. **编排器记忆是关键**：移除编排器记忆后性能显著下降
2. **联合分配最优**：编排器+Agent记忆的组合产生最强结果
3. **仅Agent记忆不足**：局部记忆缺乏全局协调，效果远不如编排器记忆
4. **即使仅在规划阶段使用编排器记忆，仍优于仅Agent记忆**

### 8.2 任务Agent专用记忆设置下的变体对比

当**仅给任务Agent提供记忆**（移除编排器记忆）时：
- LEGOMem-Dynamic和QueryRewrite在Hybrid团队中**优于Vanilla 4-5%**
- 说明**细粒度子任务检索**在编排器支持较弱时对任务Agent更有价值

### 8.3 推理增强对记忆的影响

| 设置 | LLM团队总体 | Hybrid团队总体 |
|------|-----------|--------------|
| 无推理 | 56.36 | 49.78 |
| 有推理 | 58.44 | 48.03 |

**结论**：差异微小（<2%），LEGOMem的模块化结构本身已提供足够的程序性指导。

### 8.4 执行效率和失败率

#### 执行步骤数（LLM团队）

| 任务级别 | 无记忆 | LEGOMem | 减少 |
|---------|--------|---------|------|
| Level 1 | 16.6 | 14.0 | -15.7% |
| Level 2 | 21.4 | 15.2 | -29.0% |
| Level 3 | 26.5 | 22.2 | **-16.2%** |

#### 步骤失败率（LLM团队）

| 任务级别 | 无记忆 | LEGOMem | 减少 |
|---------|--------|---------|------|
| Level 1 | 0.187 | 0.105 | -43.9% |
| Level 2 | 0.220 | 0.090 | -59.1% |
| Level 3 | 0.275 | 0.225 | **-18.2%** |

> **LEGOMem不仅提升任务成功率，还使执行更高效和可靠**。

## 十、核心贡献总结

### 10.1 三大贡献

1. **框架创新**：LEGOMem是首个为多Agent LLM系统设计的模块化程序性记忆框架
   - 将记忆分解为全任务和子任务组件
   - 角色感知的灵活分配

2. **系统性研究**：以LEGOMem为镜头，探索多Agent记忆中三个关键问题
   - **记忆应放在哪里**：编排器记忆对规划至关重要
   - **如何检索**：三种变体各有优势，Vanilla最简洁高效
   - **哪些Agent受益最多**：小模型Agent从程序性记忆中获益最大

3. **实验验证**：OfficeBench上跨三种团队配置的广泛实验
   - 所有变体一致优于基线（+12-13%绝对提升）
   - 小模型团队可匹配或超越无记忆的大模型团队

### 10.2 关键数据汇总

| 发现 | 数据支撑 |
|------|---------|
| 一致优于基线 | LLM +12.61%, Hybrid +12.72%, SLM +13.38% |
| 小模型受益最大 | SLM+LEGOMem (38.16%) > Hybrid无记忆 (35.31%) |
| 编排器记忆关键 | 移除后性能显著下降（LLM: 58.44→49.78） |
| 执行更高效 | Level 3步骤减少16.2% |
| 失败率更低 | Level 3失败率从0.275降至0.225 |
| 细粒度检索对弱Agent有价值 | Dynamic/QueryRewrite在仅Agent记忆时优于Vanilla 4-5% |

## 十一、局限性

1. **仅从成功轨迹学习**：未探索从失败轨迹中学习（未来方向）
2. **记忆整合简单**：未涉及记忆合并、遗忘等高级管理策略
3. **评估数据集有限**：仅在OfficeBench上验证
4. **固定团队架构**：基于Magentic-One风格的编排器+任务Agent架构，未探索更灵活的多Agent拓扑

---

## 十二、未来方向

1. **从失败轨迹中持续学习**
2. **扩展到开放式环境和工具生态系统**
3. **更复杂的记忆管理策略**（合并、遗忘、更新）
4. **跨团队记忆迁移**（强团队→弱团队）

### LEGOMem的独特定位

> **唯一专为多Agent系统设计的程序性记忆框架**，核心创新在于：
> 1. **模块化分解**：全任务记忆（规划层）+ 子任务记忆（执行层）
> 2. **角色感知分配**：不同角色接收不同粒度的记忆
> 3. **编排器记忆的关键性**：实验证明编排器记忆是多Agent协调的核心

**核心洞见**：在多Agent系统中，**记忆放在哪里**比**记忆内容本身**可能更重要——编排器级的全局规划记忆对系统性能的贡献远超Agent级的局部执行记忆。
