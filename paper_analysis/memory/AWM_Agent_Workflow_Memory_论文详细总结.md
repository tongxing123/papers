## 一、研究背景与动机

### 1.1 当前LLM Agent的局限

尽管LLM Agent在网页导航、移动应用操作等数字任务中展现出巨大潜力，但在面对**长时域任务（long-horizon tasks）** 和复杂动作轨迹时仍然困难重重：

| 问题 | 描述 |
|------|------|
| **缺乏鲁棒性** | 通过训练或上下文学习整合固定示例，对任务上下文或环境变化缺乏鲁棒性 |
| **无法提取可复用模式** | 未能掌握解决复杂任务的关键——**提取和学习跨相似任务共享的可复用任务工作流** |
| **无法从经验中学习** | 每个任务独立解决，不能从过去的成功和失败中学习，**无法随时间适应** |

### 1.2 人类认知的启示

人类能够灵活解决复杂任务，关键在于：
- 从过去经验中**学习可复用的任务工作流（reusable task workflows）**
- 用这些工作流**指导未来的行动**

> 例如：人类学会了"搜索商品"这一通用技能后，可以灵活应用于在不同商店搜索不同商品。

### 1.3 核心目标

> **构建能够类似地从经验中归纳常用工作流，并选择性地将其提供给Agent以指导后续生成的记忆机制。**

## 二、AWM 方法详解

### 2.1 问题形式化

考虑一个具有语言模型骨干 L 和文本记忆 M 的智能体：

- **基础记忆**：包含内置动作（如 CLICK、TYPE）的文档
- **任务求解**：对于自然语言指令 q，智能体在环境（由转移函数 T 定义）中行动
- **时间步 tᵢ**：环境状态 sᵢ 给出观察 oᵢ → 模型生成动作 aᵢ：L(q, M, oᵢ) → aᵢ
- **状态转移**：动作在环境中执行，T(sᵢ, aᵢ) → sᵢ₊₁
- **终止条件**：模型预测停止动作 aᵢ = STOP 或达到最大步数

每个完成的任务形成一个**经验 e**：
- e = (q, P_e)，其中 P_e = (p₁ᵉ, ..., pₙᵉ) 是动作轨迹
- 每步 p = (o, a)：观察 + 动作

**目标**：从经验集 E = {e} 中归纳有用的工作流 W = {w}，使用归纳模块 I：I(E) → W

### 2.2 工作流表示

每个工作流 w 由两个组件组成：

#### 组件1：工作流描述 d（Workflow Description）
- 自然语言任务描述，总结工作流的功能/目标
- 通过启发式提取或LM摘要获得
- 例："This workflow aims to find a customer order with specified ID."

#### 组件2：工作流轨迹（Workflow Trajectory）
一系列步骤 (p₁, p₂, ...)，每个步骤包含三部分：

| 部分 | 描述 | 示例 |
|------|------|------|
| **(1) 环境状态描述** | 当前环境的自然语言描述 | "Order {id} is shown" |
| **(2) 推理过程** | 智能体决定采取哪个动作的推理 | "Order {id} is found, I will now terminate" |
| **(3) 动作** | 在环境上执行的可执行程序 | `stop()` |

### 2.3 工作流归纳（Workflow Induction）

#### LM-Based 工作流归纳

核心思想：提示LM从一个或多个输入经验中**提取常见的子例程（sub-routines）**

**关键设计决策**：

1. **更细粒度的归纳**：不是归纳整个任务（"在Amazon上购买干猫粮并送到我的地址"），而是归纳更细粒度的子任务（"在Amazon上搜索商品"）
2. **上下文抽象化**：不给具体值（"干猫粮"），而是用变量名替换（`{product-name}`），**增强工作流的泛化性**
3. **分段存储**：工作流根据模型输出中的双换行符分段，分别存储在工作流记忆中

**归纳提示模板**：
```
Given a list of web navigation tasks, your task is to extract the common workflows.
Each given task contains a natural language instruction, and a series of actions.
You need to find the repetitive subset of actions across multiple tasks,
and extract each of them out as a workflow.
Each workflow should be a commonly reused sub-routine.
Do not generate similar or overlapping workflows.
Each workflow should have at least two steps.
Represent the non-fixed elements with descriptive variable names.
```

### 2.4 两种工作模式

#### Offline 模式（离线）

当有额外的高质量标注经验可用时（人类标注或模型合成）：

```
训练示例 → 拼接为单个prompt → LM生成工作流集合 W_offline
                                        ↓
测试时 → 将所有工作流整合到Agent记忆 → L(q, M + W_offline, o_test) → a_test
```

- 工作流在测试前**完全归纳**
- 所有测试使用**相同的工作流记忆** W_offline

#### Online 模式（在线）

无额外标注经验，仅使用测试查询的**无监督设置**：

```
对每个测试查询 q_t (流式处理):
  1. Agent尝试解决任务 → 生成动作轨迹 → 形成经验 e_t
  2. 评估器 L_eval(e_t) ∈ {0,1} 判断是否成功
  3. 如果成功 (1):
     → 将 e_t 转化为工作流 I(e_t) → {w_t}
     → 添加到Agent记忆: M_t + {w_t} → M_{t+1}
  4. 继续处理下一个测试查询
```

- **迭代归纳-整合-利用循环**
- 无需任何监督，仅依赖评估器判断成功/失败
- **雪球效应**：工作流随时间不断增长和复杂化

### 2.5 工作流的渐进复杂化

AWM 的一个核心特性是能够**在早期工作流基础上构建更复杂的工作流**：

```
早期阶段: "Find a place by its name"
  ├── fill('145', {location})
  ├── click('147')
  └── send_msg_to_user(...)

后期阶段: "Get the zip code of a place"  (建立在上述工作流之上)
  ├── [采用"Find a place by its name"的前几步]
  ├── fill('145', {location})
  ├── click('147')
  └── send_msg_to_user("The zip code is {zip-code}")
```

这种持续学习机制创造了**雪球效应**——随着仅处理数十个示例，与基线的性能差距可高达 **22.5个百分点**。

## 三、实验设置

### 3.1 基准数据集

| 基准 | 任务数 | 网站/域数 | 特点 | 评估方式 |
|------|:---:|:---:|------|---------|
| **WebArena** | 812 | 5个网站 | 电商、社交论坛、协作开发、内容管理 | 严格的**执行-based评估**（功能正确性） |
| **Mind2Web** | 2000+ | 200+域 | 跨任务、跨网站、跨域设置 | 元素准确率、动作F1、步骤成功率、任务成功率 |

### 3.2 评估指标

**WebArena**：
- **Task Success Rate (SR)**：任务是否功能正确完成
- **# Steps**：完成任务的平均步数

**Mind2Web**：
- **Element Accuracy**：是否正确选择了页面元素
- **Action F1**：在元素上采取的动作是否正确
- **Step SR**：元素+动作都正确
- **Task SR**：所有步骤都正确

### 3.3 基线方法

| 基线 | 类型 | 描述 |
|------|------|------|
| **BrowserGym** | 自主Agent | 当前SOTA（无人工标注站点特定知识） |
| **AutoEval** | 自主Agent | 需要额外评估和细化步骤 |
| **SteP** | 人工工作流 | 使用14个**人类专家编写的工作流** |
| **MindAct** | 自主Agent | 网页元素过滤+多选任务格式 |
| **Synapse** | 自主Agent | 轨迹风格+检索相关示例 |

### 3.4 模型配置

- **WebArena**: GPT-4 (gpt-4-0613), temperature=0.0
- **Mind2Web**: GPT-3.5-turbo 和 GPT-4, temperature=0.0
- **网页表示**: 可访问性树（accessibility trees）

## 四、实验结果与详细分析

### 4.1 WebArena 主要结果

| 方法 | 总SR | Shopping | CMS | Reddit | GitLab | Maps | #Steps |
|------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **人工工作流** |  |  |  |  |  |  |  |
| SteP (人工) | 33.0 | 37.0 | 24.0 | 59.0 | 32.0 | 30.0 | - |
| **自主Agent** |  |  |  |  |  |  |  |
| WebArena | 14.9 | 14.0 | 11.0 | 6.0 | 15.0 | 16.0 | - |
| AutoEval | 20.2 | 25.5 | 18.1 | 25.4 | 28.6 | 31.9 | 46.7 |
| BrowserGym | 23.5 | - | - | - | - | - | - |
| BrowserGym(ax-tree) | 15.0 | 17.2 | 14.8 | 20.2 | 19.0 | 25.5 | 7.9 |
| **AWM (Ours)** | **35.5** | **30.8** | **29.1** | **50.9** | **31.8** | **43.3** | **5.9** |

**关键发现**：

1. **AWM 达到 WebArena 上已发布的最佳结果**
   - 超越 BrowserGym 基线 **12.0 绝对点**，**51.1% 相对提升**
   - 甚至超越使用人类专家编写工作流的 SteP（**+7.6% 相对提升**）

2. **各网站一致提升**
   - Shopping: +13.6, CMS: +14.3, Reddit: +30.7, GitLab: +12.8, Maps: +17.8

3. **高效轨迹**
   - 比 BrowserGym 少 **2.0步/任务**
   - 比 AutoEval 少 **40.8步/任务**

### 4.2 快速学习能力

在 WebArena Map 测试集上的在线学习曲线：

```
累积成功率随处理示例数增长:

快速学习阶段 (0-40个示例):
  获取最基本的工作流 → 成功率快速上升

稳定推理阶段 (40+个示例):
  学习更高级的工作流 → 成功率逐渐稳定在最高点

关键: AWM仅需约40个示例即可实现高效学习
```

### 4.3 跨模板泛化

为验证AWM不仅仅是学习"同模板"的工作流，提取了**非重叠模板**的子集：

| 方法 | 总SR | Shopping(51) | CMS(45) | Reddit(24) | GitLab(45) | Maps(32) |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| SteP (人工) | 32.1 | 26.5 | 29.3 | 52.2 | 27.3 | 36.4 |
| AutoEval | 23.2 | 12.2 | 17.1 | 21.7 | 31.8 | 36.4 |
| BrowserGym(ax-tree) | 20.5 | 10.4 | 17.8 | 23.1 | 27.3 | 28.6 |
| **AWM (Ours)** | **33.2** | **24.5** | **29.3** | **52.2** | **31.8** | **39.4** |

**结论**：AWM归纳的工作流能**有效泛化到不同任务**（来自不同任务模板的示例）。

### 4.4 Mind2Web Offline 结果

| 方法 | Elem Acc | Action F1 | Step SR | Task SR |
|------|:---:|:---:|:---:|:---:|
| MindAct (GPT-3.5) | 20.3 | 56.6 | 17.4 | 0.8 |
| Synapse (GPT-3.5) | 34.0 | - | 30.6 | 2.4 |
| **AWM (GPT-3.5)** | **39.0** | **52.8** | **34.6** | **2.8** |
| MindAct (GPT-4) | 41.6 | 60.6 | 36.2 | 2.0 |
| **AWM (GPT-4)** | **50.6** | 57.3 | **45.1** | **4.8** |

**关键发现**：
- **Step SR**: +4.0~+8.9% 相对提升
- **Task SR**: +0.4~+2.8 绝对点提升
- 提升主要来自**更准确的元素选择**（+5.0~+9.0 元素准确率）

**抽象子例程 vs 具体经验**：
- Synapse检索最相关的完整训练示例 → 可能使Agent偏向选择类似元素
- AWM通过**抽象示例特定上下文** → 减少元素选择偏差，实现更高步骤成功率

### 4.5 Mind2Web Online 泛化结果

| 方法 | 跨任务 Step SR | 跨网站 Step SR | 跨域 Step SR |
|------|:---:|:---:|:---:|
| MindAct | 36.2 | 30.1 | 18.6 |
| AWM offline | 45.1 (+8.9) | 33.7 (+3.6) | 32.6 (+14.0) |
| **AWM online** | **43.6** (+7.4) | **33.9** (+3.8) | **35.5** (+16.9) |

**关键发现**：

1. **跨域差距最大时AWM优势最明显**：
   - 跨任务（同域）: +7.4~8.9
   - 跨网站: +3.6~3.8
   - 跨域: **+14.0~16.9** ← 分布差距越大，AWM优势越大

2. **AWM online 在未见过的网站/域上展现更强泛化能力**：
   - AWM online 不依赖训练数据 → 不受域差距影响
   - 性能差距随域差距增大而增大（apple→bestbuy < macys→reddit）

## 五、工作流表示的深入探索

### 5.1 子例程+抽象格式 vs 基于规则的归纳

| 方法 | WebArena SR | #Steps | Mind2Web Step SR |
|------|:---:|:---:|:---:|
| AWM (rule-based) | 35.6 | 6.3 | 43.4 |
| **AWM (LM-based)** | **35.5** | **5.9** | **45.1** |

- WebArena上两者性能相当（差距仅0.1），LM-based更高效（少0.4步）
- Mind2Web上LM-based提升 **+2.8**：抽象子例程格式比完整示例轨迹更灵活、更可复用

### 5.2 工作流格式：代码 vs 文本

| 方法 | Elem Acc | Action F1 | Step SR | Task SR |
|------|:---:|:---:|:---:|:---:|
| **AWM (代码)** | 50.6 | 57.3 | 45.1 | **4.8** |
| AWM (文本) | **51.2** | **57.4** | **45.4** | 3.6 |

- 两种格式**性能差异不显著**
- 文本格式步骤SR略高（+0.3），但任务SR较低（-1.2）
- **两种形式都能为Agent记忆带来有效增强**

### 5.3 环境抽象：NL描述 vs HTML

| 描述 | HTML | Elem Acc | Act F1 | Step SR | Task SR |
|:---:|:---:|:---:|:---:|:---:|:---:|
| ✓ | ✗ | **39.0** | 52.8 | **34.6** | **2.8** |
| ✗ | ✓ | 38.1 | **54.0** | 33.8 | 2.8 |
| ✓ | ✓ | 37.1 | 51.3 | 32.9 | 2.0 |

- **NL描述比HTML更有用**（替换后Step SR降0.8）
- **同时使用NL+HTML反而更差**：
  - 上下文长度显著增加 → 模型处理更困难
  - 过滤后的HTML仍有大量不相关项（47%时候遗漏所有正确元素）→ 与NL描述矛盾

## 六、工作流利用方式探索

### 6.1 工作流作为动作空间扩展（AWM_AS）

除了将工作流整合到Agent记忆中，还将工作流包装为**高层函数**，类似智能体可调用的快捷工具：

```
原始动作空间 P: {click, type, scroll, ...}
扩展动作空间: P + W = {click, type, ..., find_place, get_zipcode, ...}

调用 find_place("Central Park") → 自动执行预定的动作序列
```

| 方法 | Elem Acc | Action F1 | Step SR | Task SR |
|------|:---:|:---:|:---:|:---:|
| MindAct | 41.6 | 60.6 | 36.2 | 2.0 |
| AWM | 50.6 | 57.3 | 45.1 | 4.8 |
| **AWM_AS** | **51.8** | 56.7 | **46.4** | 3.6 |

- 步骤SR提升+1.3，但任务SR相同（3.2 vs 4.8）
- 智能体仅在 **18.5%** 的任务中调用工作流动作 → **对新增动作的抗拒**
- 动作空间扩展作为辅助动作有小幅额外收益

### 6.2 工作流动作的局限

**动态环境变化的挑战**（以航班预订为例）：

```
输入"New York" → 系统弹出附近机场选项 → 需要选择
                                              ↓
问题: 工作流动作执行预定的动作序列时，
      无法看到中间状态的弹出选项，
      不够灵活以适应动态变化
```

**潜在解决方案**：实时状态访问、动态执行循环

## 七、工作流质量分析

| 指标 | WebArena | Mind2Web | 描述 |
|------|:---:|:---:|------|
| **# Workflows** | 7.4 | 7.3 | 每个示例平均归纳的工作流数（越少越好） |
| **Coverage** | - | 0.40 | 工作流覆盖测试示例步骤的比例 |
| **Function Overlap** | 0.08 | 0.20 | 工作流对之间的功能重叠（越少越好） |
| **Utility Rate** | 0.94 | 0.91 | 工作流被测试示例使用的频率 |

**关键发现**：
- 归纳高效：平均每个示例仅7.3-7.4个工作流
- 高实用性：94%（WebArena）和91%（Mind2Web）的测试示例使用了工作流
- 低功能重叠：仅0.08-0.20，工作流各自负责不同功能
- Mind2Web覆盖率较低（0.40）是合理的，因为训练和测试示例存在显著的任务分布差异

## 八、AWM Offline + Online 整合

| 方法 | 跨任务 Step SR | 跨网站 Step SR | 跨域 Step SR |
|------|:---:|:---:|:---:|
| MindAct | 36.2 | 30.1 | 18.6 |
| AWM offline | 45.1 | 33.7 | 32.6 |
| AWM online | 43.6 | 33.9 | 35.5 |
| **AWM off+on** | 44.5 | 33.3 | 34.1 |

**发现**：AWM off+on 得分介于 offline 和 online 之间，**并非简单叠加效应**：
- 离线和在线归纳的工作流**不完全兼容**
- 离线工作流似乎**损害了在线工作流的生成质量和使用效果**

### 三大核心贡献

1. **Agent Workflow Memory (AWM)**：一种从Agent轨迹中归纳常用例程并选择性提供给Agent以指导后续生成的方法
2. **灵活的双模式运行**：
   - **Offline**：从训练示例中预先归纳工作流
   - **Online**：从测试查询中实时归纳工作流（无监督）
3. **全面的实验验证**：在WebArena和Mind2Web两大基准上，跨1000+任务、200+域验证了AWM的有效性和泛化能力

### 关键特性

| 特性 | 描述 |
|------|------|
| **细粒度子例程归纳** | 不是完整任务，而是提取可复用的子任务级别工作流 |
| **上下文抽象化** | 用变量名替换具体值，增强泛化性 |
| **渐进复杂化** | 早期简单工作流成为后期复杂工作流的子步骤（雪球效应） |
| **无监督在线学习** | 仅需评估器判断成功/失败，无需标注数据 |
| **跨域强泛化** | 分布差距越大，AWM优势越明显 |
| **高效轨迹** | 更少步数达到更高成功率 |
| **即插即用** | 作为记忆增强模块，无需修改原始Agent框架 |
