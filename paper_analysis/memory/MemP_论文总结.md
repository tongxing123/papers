## 一、研究背景与动机

### 1.1 LLM Agent面临的核心挑战

LLM Agent在执行复杂、长程任务时面临严峻困难：

- **执行代价高昂**：复杂任务需要数十个步骤和长时间运行
- **环境不可预测**：网络故障、UI变化、数据模式变更都可能中断整个流程
- **重复探索浪费**：每次失败后从头开始，但许多复杂任务在表面多样性之下具有**深层结构共性**

### 1.2 程序性记忆的缺失

> **程序性记忆（Procedural Memory）**是将习惯性技能无声地编译为可执行子程序的记忆类型，使人类能够无意识地、流畅地执行动作。

当前LLM Agent的程序性知识存在两种问题：
1. **手工编制**：以脆弱的提示模板形式存储
2. **隐式纠缠**：嵌入在模型参数中，更新代价昂贵

现有框架（LangGraph、AutoGPT、Memory Bank、Soar）提供了粗粒度抽象，但**对程序性记忆的生命周期操作（如何构建、索引、修补、修剪）几乎未做研究**。

### 1.3 本文的核心贡献

**MemP**是一个**任务无关的框架**，将程序性记忆视为**一等优化对象**，系统性地探索记忆的**构建（Build）**、**检索（Retrieve）**和**更新（Update）**策略。

## 二、相关工作定位

### 2.1 语言Agent中的记忆

记忆是语言Agent的基础组件，包含多种类型：
- **短期记忆**
- **情景记忆（Episodic Memory）**
- **长期记忆**

当前方法包括：端到端记忆系统、外部记忆系统、层次化记忆结构。

### 2.2 从经验中学习

LLM Agent从经验中学习的方法包括：
- 强化学习
- 经验回放
- 模仿学习
- 记忆管理
- 多Agent学习

**现有局限**：低样本效率、跨任务泛化差、灾难性遗忘、**记忆更新特性不足**。

### 2.3 与本文的关键区别

> MemP的核心区别在于**系统性地研究**Agent程序性知识的构建、检索和更新模块的最优策略，并在更新阶段通过维护可编辑的程序性知识库来增强Agent能力。

**相关工作对比**：
| 方法 | 与MemP的关系 |
|------|-------------|
| Voyager | 利用程序性记忆增强能力，但缺乏系统性分析 |
| AWM (Agent Workflow Memory) | 利用程序性记忆，但未深入更新机制 |
| AutoManual | 类似思路，但缺少跨轨迹任务分析 |

## 三、理论基础与问题建模

### 3.1 MDP建模

Agent与环境的交互被建模为**马尔可夫决策过程（MDP）**：

- 在每个时间步 $t$，Agent处于状态 $s_t \in S$，根据策略 $\pi(a_t|s_t)$ 选择动作 $a_t \in A$
- 环境转移到新状态 $s_{t+1}$ 并发出观察 $O_t$
- 完整交互轨迹：$\tau = (s_0, a_0, o_1, s_1, a_1, o_2, ..., s_T)$

**奖励函数**评估任务完成情况：

$$r = R(\text{env}, s_T, \tau) \in [0, 1]$$

### 3.2 程序性记忆的形式化定义

#### 记忆构建

对于任务轨迹 $\tau$ 及其奖励 $r$，由构建器 $B$ 构建记忆 $m_p$：

$$Mem = \sum_{t=1}^{T} mp_t, \quad \text{其中} \quad mp_t = B(\tau_t, r_t)$$

#### 记忆检索

面对新任务 $t_{new}$，检索最相似的记忆：

$$m_{retrieved} = \arg\max_{mp_i \in Mem} S(t_{new}, t_i)$$

使用余弦相似度：

$$m_{retrieved} = \arg\max_{mp_i \in Mem} \frac{\phi(t_{new}) \cdot \phi(t_i)}{\|\phi(t_{new})\| \|\phi(t_i)\|}$$

#### 记忆更新

更新机制建模为函数 $U$：

$$M(t+1) = U(M(t), E(t), \tau_t)$$

其中 $E(t)$ 封装执行反馈（成功/失败/性能指标），综合更新公式：

$$U = \text{Add}(M_{new}) \ominus \text{Del}(M_{obs}) \oplus \text{Update}(M_{est})$$

- **Add**：添加新的程序性记忆
- **Del**：删除过时的程序性记忆
- **Update**：基于执行反馈更新现有记忆

### 3.3 核心思想

> **从并行任务完成转向顺序任务完成**：Agent可以从早期任务中学习和蒸馏经验，减少重复探索。程序性记忆将常规策略 $\pi(a_t|s_t)$ 转化为 $\pi_{mp}(a_t|s_t)$。

## 五、实验结果：记忆存储与检索策略（Build & Retrieve）

### 5.1 记忆构建策略对比

论文设计了四种构建条件：

| 策略 | 描述 |
|------|------|
| **No Memory** | 无外部记忆，ReAct方式执行 |
| **Trajectory** | 存储完整黄金轨迹，推理时检索Top-k最相似轨迹 |
| **Script** | 分析总结黄金轨迹，蒸馏为抽象程序性知识 |
| **Proceduralization** | 结合完整检索轨迹 + 高级脚本（两者整合） |

#### 主要结果（TravelPlanner + ALFWorld）

**GPT-4o在ALFWorld上的表现**：

| 策略 | Dev成功率 | Test成功率 | 步骤数 |
|------|----------|-----------|--------|
| No Memory | 39.28% | 42.14% | 23.76 |
| Script | 66.67% | 56.43% | 18.52 |
| Trajectory | 67.17% | 74.29% | 16.49 |
| **Proceduralization** | **87.14%** | **77.86%** | **15.01** |

**关键发现**：

1. **所有记忆构建方法都优于无记忆基线**：在两个数据集上都获得更高分数并减少步骤数
2. **Script在测试集上泛化更好**：抽象脚本对不同测试任务有更强泛化能力
3. **Trajectory在开发集（相似任务）上表现更好**：完整执行轨迹更适合类似已完成任务的场景
4. **Proceduralization（两者结合）达到最优性能**：Dev提升121%，Test提升85%

### 5.2 记忆检索策略对比

论文设计了三种检索键构建方法：

| 策略 | 描述 |
|------|------|
| **Random Sample** | 不使用键进行向量检索，随机抽取记忆 |
| **Query** | 使用查询描述作为存储键，利用语义相似度检索 |
| **AveFact** | 使用大模型提取查询关键词，计算匹配关键词的平均相似度 |

#### TravelPlanner检索结果（GPT-4o）

| 策略 | #CS ↑ | #HC ↑ | Steps ↓ |
|------|-------|-------|---------|
| No Memory | 71.93 | 12.88 | 17.84 |
| Random Sample | 74.59 | 6.72 | 15.12 |
| Key=Query | 73.38 | 8.95 | 15.44 |
| **Key=AveFact** | **76.02** | **8.25** | **14.64** |

**关键发现**：

1. **精确检索显著优于随机采样**
2. **Query方法**受益于捕获语义上下文，实现更精确匹配
3. **AveFact方法**通过提取关键特征并平均相似度，有效聚焦核心任务要素
4. **语义理解和关键特征提取显著提升记忆访问准确性**

## 六、实验结果：记忆更新策略（Update）

### 6.1 三种更新策略

| 策略 | 描述 | 特点 |
|------|------|------|
| **Vanilla** | 每完成t个任务后，将所有轨迹整合为程序性记忆并直接追加 | 简单粗暴，不区分成功/失败 |
| **Validation** | 只提取成功完成任务的轨迹，抽象为紧凑的程序性记忆；失败轨迹被丢弃 | 选择性整合 |
| **Adjustment** | 当检索的程序性记忆导致执行失败时，将错误轨迹与原始记忆结合并就地修正 | 基于反思的错误纠正 |

### 6.2 更新策略性能演化

论文将测试集任务分为多个组，每组完成后刷新记忆，观察性能演化：

**关键发现**：

1. **所有策略都贡献了后续任务的性能提升**：随着顺序完成更多组并迭代刷新记忆
2. **Adjustment（反思式更新）最为有效**：
   - 到最后一组任务时，超过第二好策略**+0.7分**
   - 步骤数减少**14步**
3. **持续更新记忆的价值巨大**：特别是当更新由错误纠正机制引导时

### 6.3 更新机制的数学表达

$$M(t+1) = \text{Add}(M_{new}) \ominus \text{Del}(M_{obs}) \oplus \text{Update}(M_{est})$$

这捕获了更新机制中的核心操作：
- **Add**：新经验直接入库
- **Del**：过时/冗余记忆被淘汰
- **Update**：基于执行反馈修正现有记忆

## 八、分析与洞见

### 8.1 程序性记忆提升准确率并减少尝试次数

**案例分析**（加热鸡蛋并放入垃圾桶）：

| 条件 | 步骤数 | Token消耗 | 是否完成 |
|------|--------|----------|---------|
| 无记忆（场景1） | 27 | 3,635 | ❌ 失败 |
| 无记忆（场景2） | 23 | 3,274 | ✅ 完成（但代价高） |
| **有程序性记忆** | **14** | **2,589** | ✅ 完成 |

**无记忆时的两种困境**：
1. **场景1**：模型反复尝试非法/错误动作，上下文越来越复杂，最终超出模型理解能力
2. **场景2**：多次尝试后完成任务，但代价远高于最优路径

**有程序性记忆后**：
- 记忆指出鸡蛋的大致位置，省去无目的搜索
- 加热过程中提供明确指导，确保连续正确执行
- **步骤减少9步，Token节省685个**

### 8.2 程序性记忆具有跨模型迁移性

> **强模型构建的程序性记忆可以有效迁移到弱模型**

**GPT-4o → Qwen2.5-14B 迁移结果**：

| 数据集 | 指标提升 |
|--------|---------|
| TravelPlanner | 任务完成率提升**5%**，平均步骤减少**1.6步** |
| ALFWorld | 成功率和轨迹长度均获类似提升 |

**意义**：
- 强模型的程序性知识可以蒸馏为可复用的记忆库
- 以最小开销迁移到更小模型
- 为小模型提供明确的任务解决能力提升

**GPT-4o记忆迁移到Qwen2.5-14B的详细结果**：
- Delivery评分：从0.0%提升至16.9%
- 常识评分：从59.3%提升至91.4%
- 总步骤：从96.6降至65.5

### 8.3 检索记忆数量的影响

**检索数量与性能的关系**：

- 随着检索程序性记忆数量增加，Agent性能**稳步提升**，呈上升趋势后趋于平稳
- **过度检索会导致性能下降**：
  - 影响上下文长度
  - 引入不太准确的程序性记忆
  - 干扰整体效果

**最佳平衡点**：需要适量检索以获取足够信息，同时避免噪声干扰。

## 九、框架核心设计理念

### 9.1 程序性记忆的两种存储格式

| 格式 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **原始轨迹（Trajectory）** | 逐轮保留完整交互记录 | 信息完整，适合相似任务 | 泛化性差，存储大 |
| **抽象脚本（Script）** | 从轨迹中提取高层抽象知识 | 泛化性强，紧凑 | 可能丢失细节 |
| **程序化（Proceduralization）** | 两者结合 | 兼顾完整性和泛化性 | 最复杂 |

### 9.2 记忆的生命周期

```
  构建(Build)           检索(Retrieve)          更新(Update)
 ┌──────────┐         ┌──────────┐           ┌──────────┐
 │ 任务执行  │         │ 新任务到来 │           │ 执行反馈  │
 │    ↓     │         │    ↓     │           │    ↓     │
 │ 轨迹记录  │         │ 键构建    │           │ 成功→追加  │
 │    ↓     │         │    ↓     │           │ 失败→修正  │
 │ 记忆构建  │         │ 向量匹配  │           │ 过时→删除  │
 │ (全轨迹/  │         │    ↓     │           │    ↓     │
 │  脚本/    │         │ 相关记忆  │           │ 记忆库演进 │
 │  程序化)  │         │ 注入上下文 │           │          │
 └──────────┘         └──────────┘           └──────────┘
```

## 十、核心贡献总结

### 10.1 三大核心贡献

1. **框架创新**：提出MemP，一个任务无关的框架，将程序性记忆提升为LLM Agent的核心优化对象

2. **系统性研究**：首次系统探索记忆的构建、检索和更新三大模块的最优策略
   - **构建**：Proceduralization（轨迹+脚本）效果最优
   - **检索**：AveFact（关键词平均相似度）最精确
   - **更新**：Adjustment（反思式修正）最有效

3. **重要发现**：
   - 程序性记忆同时提升准确率和效率
   - 记忆具有跨模型迁移性（强→弱）
   - 检索数量存在最优平衡点
   - 持续更新带来近线性的能力增长

### 10.2 关键实验数据汇总

| 发现 | 数据支撑 |
|------|---------|
| 程序性记忆有效 | ALFWorld Dev: 39.28% → 87.14% (+122%) |
| 程序化最优 | 全面超越Script和Trajectory单独使用 |
| 精确检索重要 | AveFact比Random提升显著 |
| 反思更新最强 | 超过第二好策略+0.7分，减少14步 |
| 跨模型迁移 | GPT-4o→Qwen2.5-14B: 完成率+5%, 步骤-1.6 |
| 效率提升 | 案例中步骤减少9步(33%), Token节省685(19%) |

## 十一、局限性

1. **检索受限**：仅限于向量相似度搜索和手工构建的键，未纳入BM25等经典精确检索方法

2. **依赖显式奖励信号**：框架依赖基准测试提供的显式奖励信号，无法在真实场景中奖励稀疏或缺失时自主判断任务成功

---

## 十二、未来方向

1. **更丰富的检索策略**：当前仅构建了不同的键用于向量检索，计划开发更多样化的检索方法

2. **LLM-as-Judge自主评估**：用LLM作为裁判评估任务完成情况，替代基准测试的显式奖励，将Agent生命周期转化为持续循环：
   > **执行任务 → 自我评估 → 构建记忆 → 持续进步**

---

## 十三、关键启示与评价

### 优势
1. **问题定义清晰**：将程序性记忆形式化为MDP框架，有坚实的理论基础
2. **实验系统全面**：三个骨干模型 × 两个数据集 × 多维度策略对比
3. **实用发现丰富**：构建/检索/更新各环节的最优策略对实际应用有直接指导意义
4. **迁移性验证**：强模型→弱模型的记忆迁移是极具价值的发现
5. **持续学习范式**：展示了Agent通过程序性记忆实现近线性能力增长的可能

### 不足
1. **实验规模有限**：仅两个数据集，多样性不够
2. **检索策略简单**：仅测试了三种键构建方法
3. **更新策略粗粒度**：Vanilla/Validation/Adjustment三种策略的区分度不够精细
4. **缺乏与其他记忆类型（如情景记忆、语义记忆）的对比**
5. **未探讨记忆的容量限制和遗忘机制**
