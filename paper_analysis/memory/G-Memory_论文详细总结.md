## 一、研究背景与动机

### 1.1 LLM Agent 的自我进化能力

LLM驱动的智能体已展现出前所未有的感知、规划、推理和行动能力。**自我进化能力（Self-Evolving Capacity）** 是LLM Agent的核心特征——通过与环境交互不断适应和改进。先前的研究表明，这种适应性可以带来**2-3倍的定量改进**。

**记忆机制**是自我进化的核心驱动力，类比人类积累知识、处理过往经验和检索相关信息的能力。成功的记忆设计包括：
- **Trial内记忆（Inside-trial Memory）**：解决单个查询时保留的上下文
- **跨Trial记忆（Cross-trial Memory）**：跨多个任务积累的经验

### 1.2 多智能体系统（MAS）的自我进化困境

尽管单智能体的自我进化已取得显著进展，**多智能体系统的自我进化能力仍然严重缺失**：

| 问题类型 | 具体表现 |
|---------|---------|
| **手动定义的工作流** | MetaGPT的SOP、ChatDev的开发流程、MacNet的预定义通信拓扑 |
| **自动化MAS的笨重** | GPTSwarm、ADAS、AFlow、MaAS等虽然自动优化拓扑/prompt，但最终产出巨大、笨重的架构 |
| **缺乏灵活性** | 无法随协作经验的积累进行自我调整 |

### 1.3 MAS记忆机制的关键缺口

**问题❶：现有MAS是否完全没有记忆机制？**

不完全是。MetaGPT、ChatDev、Exchange-of-Thought等框架包含记忆设计，但：
- 通常仅限于 **inside-trial memory**
- 跨Trial记忆（如存在）极其粗糙——仅传递过度压缩的产物（最终方案或执行结果）
- **无法从协作经验中进行有意义的学习**

**问题❷：为什么不直接将单智能体记忆迁移到MAS？**

MAS的固有特性——**多智能体间的多轮编排**——导致任务解决轨迹远长于单智能体设置（**高达10×以上的token消耗**）。这给传统基于检索的记忆设计带来巨大挑战：

```
ALFWorld基准上的Token消耗对比：
├── CoT (单智能体):      ~5.8×10⁵ tokens
├── ComplexCoT (单智能体): ~6.1×10⁵ tokens
├── AutoGen (MAS):       ~4.3×10⁶ tokens  (约7.4×)
└── DyLAN (MAS):         ~7.9×10⁶ tokens  (约13.6×)
```

**不加抽象地将整个长上下文轨迹直接喂给LLM，几乎不会带来任何好处。**

### 1.4 核心研究问题

> **如何设计一种记忆机制，能够存储、检索和管理多智能体系统的冗长交互历史，使得智能体团队能够从简洁而有指导意义的经验和洞察中获益？**

## 二、G-Memory 概述

### 2.1 核心定义

**G-Memory**（Graph-based Agentic Memory Mechanism）是为LLM多智能体系统设计的**层次化、智能体式记忆系统**，灵感来自**组织记忆理论（Organizational Memory Theory）**。

### 2.2 三层图层次架构

G-Memory 通过**三层图层次结构**管理MAS的复杂和冗长交互历史：

```
┌──────────────────────────────────────────────────────────────┐
│                  G-Memory 三层层次架构                         │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │  ✱ Insight Graph (洞察图)  —— 最高层                    │  │
│  │  · 从历史经验中抽象出可泛化的高层洞察                     │  │
│  │  · 节点: 洞察内容 + 支持查询集合                         │  │
│  │  · 边: 超连接 (洞察间通过查询产生的上下文关系)            │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │ 向上遍历 ↑ ↓ 向下遍历             │
│  ┌────────────────────────┴───────────────────────────────┐  │
│  │  ✱ Query Graph (查询图)  —— 中间层                      │  │
│  │  · 编码任务查询的元信息和查询间的连通性                   │  │
│  │  · 节点: 原始查询 + 任务状态 + 关联交互图                │  │
│  │  · 边: 查询间的语义关系                                  │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │                                   │
│  ┌────────────────────────┴───────────────────────────────┐  │
│  │  ✱ Interaction Graph (交互图)  —— 最底层                │  │
│  │  · 存储智能体间细粒度的文本通信日志                       │  │
│  │  · 节点: 原子话语 (发言智能体 + 文本内容)                │  │
│  │  · 边: 时间关系 (话语间的传递和启发关系)                  │  │
│  └────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

## 三、技术方法详解

### 3.1 预备知识：多智能体系统形式化

**MAS定义**：多智能体框架表示为有向图 G = (V, E)，其中 |V| = N 为智能体数量。

每个智能体 Cᵢ ∈ V 由四元组描述：

$$C_i = (\text{Base}_i, \text{Role}_i, \text{Mem}_i, \text{Plugin}_i)$$

| 组件 | 描述 |
|------|------|
| **Baseᵢ** | 底层LLM实例 |
| **Roleᵢ** | 智能体的指定角色/人设 |
| **Memᵢ** | 记忆状态（过去的交互或外部知识存储） |
| **Pluginᵢ** | 辅助工具集（如搜索引擎） |

**通信过程**：在每个通信轮次 t，对节点进行拓扑排序 π = [π₁, ..., πₙ]，保证每个智能体在其所有前驱行动后才处理输入。

**全局聚合**：所有智能体行动后，全局聚合算子 A 融合响应集合为临时方案 a⁽ᵗ⁾。

### 3.2 三层图结构的形式化定义

#### 3.2.1 交互图（Interaction Graph）

对于查询 Q，交互轨迹为 G_inter⁽ᵠ⁾ = (U⁽ᵠ⁾, E_u⁽ᵠ⁾)：

- **节点** U⁽ᵠ⁾ = {uᵢ}：原子话语，每个 uᵢ ≜ (Aᵢ, mᵢ)，其中 Aᵢ ∈ V（发言智能体），mᵢ（文本内容）
- **边** E_u⁽ᵠ⁾ ⊆ U⁽ᵠ⁾ × U⁽ᵠ⁾：时间关系，(uⱼ, uₖ) ∈ E_u ⇔ uⱼ 传递给 uₖ 并启发 uₖ

#### 3.2.2 查询图（Query Graph）

$$G_{query} = (\mathcal{Q}, E_q) = \left(\left\{Q_i, \Psi_i, G_{inter}^{(Q_i)}\right\}_{i=1}^{|\mathcal{Q}|}, E_q\right)$$

- **节点** qᵢ ≜ (Qᵢ, Ψᵢ, G_inter⁽ᵠᵢ⁾)：原始查询 Qᵢ、任务状态 Ψᵢ ∈ {Failed, Resolved}、关联交互图
- **边** E_q ⊆ Q × Q：查询间的语义关系

**核心价值**：查询图使得检索超越了embedding相似度等粗糙指标，利用精细的拓扑结构实现更准确的检索。

#### 3.2.3 洞察图（Insight Graph）

$$G_{insight} = (\mathcal{I}, E_i) = \left(\left\{\kappa_k, \Omega_k\right\}_{k=1}^{|\mathcal{I}|}, E_i\right)$$

- **节点** ιₖ：提炼的洞察，由洞察内容 κₖ 和支持查询集合 Ωₖ ⊆ Q 组成
- **边** E_i ⊆ I × I × Q：**超连接（Hyper-connections）**，(ιₘ, ιₙ, qⱼ) 表示洞察 ιₘ 通过查询 qⱼ 上下文化 ιₙ

### 3.3 工作流程总览

G-Memory 的管理流程分为三个阶段：

```
新查询 Q 到达
    │
    ├── 阶段1: 粗粒度记忆检索 (Section 4.1)
    │       └── 在查询图上进行相似度检索 + 跳扩展
    │
    ├── 阶段2: 双向记忆遍历 (Section 4.2)
    │       ├── 向上遍历: Query Graph → Insight Graph (获取高层洞察)
    │       └── 向下遍历: Query Graph → Interaction Graph (获取协作轨迹)
    │
    └── 阶段3: 层次记忆更新 (Section 4.3)
            ├── 更新交互图 (追踪话语)
            ├── 更新查询图 (添加新节点和边)
            └── 更新洞察图 (生成新洞察 + 更新现有洞察)
```

### 3.4 阶段1：粗粒度记忆检索（Coarse-grained Memory Retrieval）

**步骤1：相似度检索**

在查询图 G_query 上进行粗粒度的相似度检索，获取查询草图集合：

$$QS = \arg\text{top-}k_{q_i \in \mathcal{Q}, |QS|=k} \left(\frac{v(Q) \cdot v(q_i)}{|v(Q)| |v(q_i)|}\right)$$

其中 v(·) 使用 MiniLM 等模型将查询映射为固定长度embedding。

**步骤2：跳扩展（Hop Expansion）**

相似度可能仅是表面的或有噪声的，因此通过查询图上的跳扩展扩大相关集合：

$$\tilde{QS} = QS \cup \{Q_k \in \mathcal{Q} \mid \exists Q_j \in QS, Q_k \in N^+(Q_j) \cup N^-(Q_j)\}$$

即 QS 在查询图上的 **1-hop 邻居**被加入。

**为什么不能直接使用这些记录？**
- 过长的上下文可能使LLM不堪重负
- MAS中的智能体扮演不同角色，应分配**专门针对其功能的记忆**

### 3.5 阶段2：双向记忆遍历（Bi-directional Memory Traversal）

#### 向上遍历：Query Graph → Insight Graph

检索能为当前任务提供高层指导的洞察节点：

$$IS = \Pi_{Q \to I}(\tilde{QS}), \quad \Pi_{Q \to I}(S_q) \triangleq \{\iota_k \in \mathcal{I} \mid \Omega_k \cap S_q \neq \emptyset\}$$

其中 Π_{Q→I} 是查询到洞察的投影器，识别所有支持查询集与输入查询集有交集的洞察节点。

**检索到的洞察 IS**：封装了提炼的、泛化的知识，可用于指导MAS的战略方法。

#### 向下遍历：Query Graph → Interaction Graph

利用LLM辅助的图稀疏化器 S_LLM(·,·) 提取封装核心智能体间协作的子图：

$$\{\hat{G}_{inter}^{Q_i}\}_{i=1}^{|M|} = \left\{S_{LLM}(G_{inter}^{(Q_j)}, Q) \mid q_j \in \arg\text{top-}M_{q'_k \in \tilde{QS}} R_{LLM}(Q, q'_k)\right\}$$

其中：
- R_LLM(Q, qⱼ) 评估历史查询相对于 Q 的相关性
- S_LLM 从原始交互图中构建稀疏化图，识别并保留关键对话元素

#### 智能体专属记忆分配

$$\text{Mem}_i \leftarrow \Phi\left(IS, \{\hat{G}_{inter}^{Q_i}\}_{i=1}^{|M|}; \text{Role}_i, Q\right)$$

算子 Φ(·;·) 评估每个洞察和稀疏化交互图相对于智能体特定角色 Roleᵢ 和任务 Q 的效用和相关性，为每个智能体初始化专门的内部记忆状态。

### 3.6 阶段3：层次记忆更新（Hierarchy Memory Update）

任务完成后，系统接收环境反馈（执行状态 Ψᵢ ∈ {Failed, Resolved}，token使用量等），三层记忆全部更新：

#### 交互层更新
追踪每个智能体的话语，构建交互图 G_inter⁽ᵠ⁾ 并存储。

#### 查询层更新

$$q_{new} \leftarrow (Q, \Psi, G_{inter}^{(Q)})$$

$$N_{conn} \leftarrow Q_R \cup \left(\bigcup_{\iota_k \in IS} \Omega_k\right)$$

$$E_{new} \leftarrow \{(q_n, q_{new}) \mid q_n \in N_{conn}\}$$

新查询节点与以下节点建立边：
1. 方程(7)中识别的top-M相关历史查询集 Q_R
2. 支持所用洞察 IS 的查询集合

#### 洞察层更新

**生成新洞察**：

$$\iota_{new} = (J(G_{inter}^{(Q)}, \Psi), \{q_{new}\})$$

其中 J(·,·) 是总结函数，从完成的交互图和执行状态中生成新洞察。

**添加超连接边**：

$$E_{i,new} \leftarrow \{(\iota_k, \iota_{new}, q_{new}) \mid \iota_k \in IS\}$$

**更新支持查询集**：已使用洞察的 Ωₖ 更新为包含 q_new，反映其与该成功（或失败）应用的相关性。

### 5.2 关键发现

#### 发现❶：G-Memory 在所有任务领域和MAS框架上一致提升性能

- AutoGen + Qwen-2.5-7b：G-Memory 超越最佳单/多智能体记忆基线平均 **6.8%**
- DyLAN + Qwen-2.5-7b：超越最佳基线平均 **5.5%**
- MacNet + Qwen-2.5-14b 在 ALFWorld：从 58.21% → **79.10%**，提升 **20.89%**

#### 发现❷：多智能体系统需要专门的记忆设计

**大多数记忆机制无法在MAS设置中一致地带来收益**：

| 示例 | 性能退化 |
|------|---------|
| Voyager + AutoGen on PDDL | 降低 **4.17%** |
| MemoryBank + AutoGen on PDDL | 降低 **1.34%** |
| ChatDev-M + MacNet on SciWorld (Qwen-2.5-7b) | 降低 **2.32%** |
| ChatDev-M + DyLAN (Qwen-2.5-7b) | 平均降低 **4.41%** |

**原因分析**：
- 这些方法无法提供**智能体角色特定的记忆支持**
- ChatDev-M仅存储执行结果，在具身动作环境中效用有限
- G-Memory 的核心特征——**角色特定记忆线索、抽象高层洞察、轨迹压缩**——对MAS的有效记忆至关重要

#### 发现❸：G-Memory 实现高性能集体记忆而无需过度token消耗

**Token效率对比（PDDL + AutoGen）**：

| 方法 | 性能提升 | Token增量 |
|------|---------|----------|
| **G-Memory** | **+10.32%** | 仅 +1.4×10⁶ |
| MetaGPT-M | +4.07% | +2.2×10⁶ |

G-Memory 以**更少的token开销**获得**更大的性能提升**。

| **1-hop** | **~85.82%** | **~55.24%** |
| 2-hop | ~82% | ~49.79% |
| 3-hop | ~78% | ~46% |

**结论**：1-hop扩展一致产生最佳或接近最佳性能。过度跳扩展引入不相关洞察，损害任务特定推理。

#### 检索查询数 k 的影响

| k值 | ALFWorld+AutoGen | FEVER+DyLAN |
|:---:|:---:|:---:|
| **1** | **~85%** | ~64% |
| **2** | **~86%** | **~66.66%** |
| 3 | ~82% | ~65% |
| 5 | ~78% (↓7.71%) | ~64% (↓2.5%) |

**结论**：最优 k ∈ {1,2}。更大的k值引入任务无关噪声。

### 5.5 消融实验

| 变体 | AutoGen (PDDL) | AutoGen (FEVER) | DyLAN (PDDL) | DyLAN (FEVER) |
|------|:---:|:---:|:---:|:---:|
| 仅交互（无洞察） | 54.46 | 63.27 | 48.75 | 61.39 |
| 仅洞察（无交互） | 50.00 | 68.77 | 46.69 | 64.31 |
| **完整G-Memory** | **55.24** | **71.43** | **51.12** | **66.66** |

**结论**：
- 移除任一部分均导致一致的性能下降
- 仅使用交互时：AutoGen平均下降 **4.47%**，DyLAN下降 **3.82%**
- 仅使用洞察时：AutoGen平均下降 **3.95%**，DyLAN下降 **3.39%**
- **两者互补**：交互提供稍大影响（保留更多细粒度、对话级别的上下文基础），洞察提供高层泛化指导

## 九、局限性与未来方向

### 当前局限

1. **基准多样性**：虽然在3个领域5个基准上评估，但更多样化的任务（如医学QA）将进一步增强其可靠性
2. **LLM依赖性**：不同LLM生成的实体链接结构可能影响记忆操作质量

### 未来方向

1. 更多样化任务领域的验证
2. 探索更细粒度的调用策略（如每轮对话开始时或选择性为特定智能体调用）
3. 跨域迁移能力
