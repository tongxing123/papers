## 一、论文概述

本文是**首篇**系统性地综述 LLM-based Agent 记忆机制的 survey 论文。论文围绕三个核心问题展开：

1. **What is** —— Agent 的记忆是什么？
2. **Why do we need** —— 为什么 Agent 需要记忆？
3. **How to implement and evaluate** —— 如何设计与评估记忆模块？

此外，论文还展示了记忆增强的 Agent 在多个应用场景中的实践，并分析了当前研究的局限性与未来方向。

## 二、核心概念定义

### 2.1 基本定义

| 概念 | 定义 |
|------|------|
| **Agent（智能体）** | 能够与环境交互的智能实体，可获取观察和反馈，进行思考和决策，采取行动影响环境 |
| **Task（任务）** | Agent 需要达成的最终目标，如为 Alice 订机票、为 Bob 推荐餐厅等 |
| **Environment（环境）** | 狭义上是 Agent 完成任务所需交互的对象（用户、网站等）；广义上包含 Agent 可感知的一切 |
| **Trial（尝试）** | 从 Agent 开始执行任务到完成任务的一个完整交互过程 |
| **Step（步骤）** | Trial 中 Agent 与环境的一次交互（最细粒度单元） |

### 2.2 记忆的狭义定义（Narrow Definition）

仅包含**同一 trial 内**的历史信息。形式化表示：

$$b_C = \{o_1, a_1, o_2, a_2, \dots, o_{C-1}, a_{C-1}\}$$

其中 $o_i$ 为环境观察，$a_i$ 为 Agent 行动，$C$ 为当前步骤。

### 2.3 记忆的广义定义（Broad Definition）

记忆信息来自三个来源：

1. **同一 trial 内的历史信息** $b_C^j$
2. **跨 trial 的历史信息** $\mathcal{B}^j = \{b_1, b_2, \dots, b_{j-1}, b_{j'}\}$（包括之前任务的历史 trial 和当前任务的已探索 trial）
3. **外部知识** $\mathcal{K}_C^j$（超出 Agent-环境交互的知识）

### 2.4 三大记忆操作

论文给出了一个统一的形式化表达：

| 操作 | 功能 | 公式 |
|------|------|------|
| **Memory Writing（写入）** | 将感知到的信息存储到记忆中 | $M_C^j = f(\{o_C^j, e_C^j\})$ |
| **Memory Management（管理）** | 对已存储信息进行处理，使其更有效 | $M_C^j = g(M_{C-1}^j, m_C^j)$ |
| **Memory Reading（读取）** | 从记忆中提取有用信息支持下一步行动 | $\hat{M}_C^j = h(M_C^j, c_{C+1}^j)$ |

统一演化公式：

$$o_{C+1}^j = \text{LLM}\{h(g(M_{C-1}^j, f(\{o_C^j, e_C^j\})), c_{C+1}^j)\}$$

## 三、为什么需要记忆？

论文从三个视角论证了记忆对 LLM-based Agent 的必要性：

### 3.1 认知心理学视角

- 记忆是人类认知过程中极其重要的组成部分
- Agent 的目标是替代人类完成任务，因此模拟人类的记忆机制是自然且必要的选择
- 认知心理学长期积累的记忆理论可以为 Agent 设计提供指导

### 3.2 自我演化视角

记忆对 Agent 的自演化至关重要：

1. **经验积累**：记住过去的错误规划和失败经验，提升未来处理类似任务的效率
2. **环境探索**：基于历史信息决定何时、如何进行探索
3. **知识抽象**：从原始观察中总结和抽象高层信息，增强 Agent 的适应性和泛化能力

### 3.3 应用视角

在许多应用场景中，记忆不可或缺：
- **个人助理**：需要记住用户的偏好和习惯
- **角色扮演**：需要维持角色的一致性
- **代码生成**：需要记住项目上下文和开发需求
- **游戏**：需要积累技能和经验

## 四、如何实现记忆？

论文从三个维度系统梳理了记忆的实现方法：**记忆来源**、**记忆形式**、**记忆操作**。

### 4.1 记忆来源（Memory Sources）

#### 4.1.1 Trial 内信息（Inside-trial Information）

最直接和直观的记忆来源，几乎所有工作都使用此信息。

**代表工作**：
- **Generative Agents**：从历史行为中推导记忆
- **MemoChat**：基于对话会话历史构建记忆
- **Voyager**：将可执行代码存储为记忆（Minecraft 游戏）
- **Gödel Agent**：包含 Agent 自身代码、环境反馈、历史操作记录

**局限**：仅依赖 trial 内信息会阻止 Agent 从不同任务中积累可泛化的知识。

#### 4.1.2 跨 Trial 信息（Cross-trial Information）

跨多次 trial 积累的信息，包括成功/失败的动作及其洞察。

**代表工作**：
- **Reflexion**：提出语言强化学习，从过去 trial 中获取经验并应用于后续 trial
- **ExpeL**：存储已完成的轨迹，召回相似轨迹，通过比较成功/失败案例识别成功模式
- **Synapse**：记录跨 trial 的成功范例作为参考
- **AWM**：从历史轨迹中生成可复用的工作流

**优势**：可视为长期记忆，提供更广泛的经验支持。

#### 4.1.3 外部知识（External Knowledge）

通过 API 调用工具获取外部知识（如 Wikipedia、OpenWeatherMap 等）。

**代表工作**：
- **Toolformer**：教 LLM 使用工具获取外部知识
- **ToolLLM**：赋予 LLM 使用 RapidAPI 中 16000+ API 的能力
- **ReAct**：结合推理和行动，通过工具获取外部知识

**三种来源的比较**：

| 来源 | 泛化性 | 相关性 | 可解释性 |
|------|--------|--------|----------|
| 外部知识 | 最强（共享知识） | 较弱 | 取决于存储方式 |
| 跨 Trial 信息 | 中等（场景内泛化） | 中等 | 较好（文本形式） |
| Trial 内信息 | 最弱（限于当前轨迹） | 最强 | 最好（文本形式） |

---

### 4.2 记忆形式（Memory Forms）

#### 4.2.1 文本形式（Textual Form）

当前主流方法，特点是可解释性好、实现简单、读写效率高。包含四种子类型：

| 子类型 | 描述 | 优势 | 劣势 |
|--------|------|------|------|
| **完整交互记录** | 存储所有 Agent-环境交互历史 | 信息完整 | 计算成本高、推理延迟大、可能超出上下文窗口 |
| **近期交互记录** | 只保留最近的记忆（类似缓存/滑动窗口） | 效率高、聚焦近期信息 | 可能丢失远距离关键信息 |
| **检索到的交互记录** | 基于相关性/重要性检索记忆 | 能召回远距离但重要的记忆 | 检索质量依赖索引设计 |
| **外部知识** | 通过工具调用获取外部信息 | 知识边界无限、信息实时更新 | 可靠性存疑、隐私安全问题 |

**代表工作**：
- 完整交互：**SCM**（闪存机制）、**ChatDev**（聊天链）
- 近期交互：**MemGPT**（将 Agent 视为操作系统）、**RecAgent**（短期记忆缓存）
- 检索交互：**Generative Agents**（重要性+时效性+相关性评分）、**RecMind**（自启发式检索）

#### 4.2.2 参数形式（Parametric Form）

将信息编码到模型参数中，隐式影响 Agent 行为。

**微调方法（Fine-tuning）**：
- 通过监督微调将领域知识注入 LLM
- **优势**：能有效弥合通用 Agent 与专业 Agent 的差距
- **劣势**：可能过拟合、灾难性遗忘、计算成本高、不适合在线场景

**知识编辑方法（Memory Editing）**：
- 针对特定事实进行精确调整，不影响无关知识
- **代表工作**：**MAC**（元学习框架）、**MEND**（超网络生成参数修改）、**PersonalityEdit**（编辑 Agent 人格）
- **优势**：计算成本低、适合在线场景
- **劣势**：适合小规模记忆调整

#### 4.2.3 两种形式的比较

| 维度 | 文本形式 | 参数形式 |
|------|----------|----------|
| 容量 | 受上下文窗口限制 | 理论上无限制 |
| 写入效率 | 高（尤其小量数据） | 低（需要训练/编辑） |
| 读取效率 | 低（需拼接到 prompt） | 高（已融入参数） |
| 可解释性 | 高（自然语言） | 低（潜在空间表示） |
| 信息密度 | 低（离散空间） | 高（连续空间） |

---

### 4.3 记忆操作（Memory Operations）

#### 4.3.1 记忆写入（Memory Writing）

决定哪些信息需要存储。

**策略**：
- **存储原始信息**：直接保存 Agent-环境交互的原始内容
- **存储摘要信息**：对原始信息进行总结后存储（如 MemoChat 对对话段进行主题抽象）
- **自主写入**：Agent 根据上下文自主决定更新记忆（如 MemGPT）
- **结构化写入**：将信息组织为关系、树等结构（如 TiM 将信息提取为实体关系存入数据库、MemTree 用树结构组织记忆）

#### 4.3.2 记忆管理（Memory Management）

对已存储信息进行处理，包含三种子操作：

| 操作 | 描述 | 代表工作 |
|------|------|----------|
| **合并（Merging）** | 将相似信息合并减少冗余 | RET-LLM（相似轨迹合并）、AriGraph（知识图谱更新） |
| **反思（Reflection）** | 从历史记忆中生成高层抽象信息 | Generative Agents（累积事件触发反思）、GITM（总结关键动作） |
| **遗忘（Forgetting）** | 移除不重要或不相关的信息 | AriGraph（移除过时边和顶点）、MEOW（通过矛盾事实实现敏感信息遗忘） |

#### 4.3.3 记忆读取（Memory Reading）

从记忆中提取与当前状态相关的信息。

**方法**：
- **SQL 查询**：ChatDB 通过生成 SQL 语句读取记忆
- **向量检索**：ExpeL 使用 FAISS 向量库获取 top-k 最相似的成功轨迹
- **结构化检索**：AriGraph 同时进行语义搜索（三元组）和情景搜索（情景顶点）
- **树结构检索**：MemTree 使用折叠树检索方法

## 五、如何评估记忆？

论文总结了两大类评估策略：

### 5.1 直接评估（Direct Evaluation）

将记忆作为独立组件进行评估。

#### 主观评估

| 评估维度 | 描述 |
|----------|------|
| **连贯性（Coherence）** | 召回的记忆是否自然且适合当前上下文 |
| **合理性（Rationality）** | 召回的记忆是否合理 |

**关键问题**：评估者选择（需熟悉任务且背景多样）、标注方式（直接打分 vs 对比排序）

#### 客观评估

| 评估指标 | 描述 | 公式 |
|----------|------|------|
| **结果正确性** | Agent 基于记忆给出的答案是否正确 | $\frac{1}{N}\sum I(y_i = \hat{y}_i)$ |
| **参考准确性** | Agent 是否找到了相关的记忆内容来支持决策 | F1-score |
| **时间与硬件成本** | 记忆适配和推理的时间消耗 | $\frac{1}{N}\sum (t_i^{end} - t_i^{start})$ |

### 5.2 间接评估（Indirect Evaluation）

通过任务完成质量来反映记忆效果。

| 任务类型 | 评估方式 | 代表指标 |
|----------|----------|----------|
| **对话** | 一致性与参与度 | GPT-4 评分、SCE-p 分数、CSIM 分数 |
| **多源问答** | 综合评估多来源记忆的整合利用 | 准确率 |
| **长上下文应用** | 评估长文本理解和摘要能力 | ROUGE 等匹配分数 |
| **其他任务** | 成功率、探索程度 | 任务完成率、探索的独特物品数量 |

### 5.3 数据集与基准

| 数据集/基准 | 特点 |
|-------------|------|
| **PerLTQA** | 个性化记忆评估，关注事件和社会连接 |
| **LeMon / DuLeMon** | 长期记忆对话，关注历史信息理解和记忆 |
| **MSC** | 多会话聊天数据集 |
| **MemDaily** | 包含多种问题类型，关注效果和效率 |
| **LongMemEval** | 全面评估聊天助手记忆能力（IE、多会话推理、时间推理、知识更新、弃权） |

### 5.4 性能对比分析

**效果视角**：
- 长期场景：检索方法表现最佳（LongMemEval 中 >50% 准确率）
- 完整交互方法在 MemDaily 中可达 >80% 准确率，但依赖长上下文支持
- 近期信息方法的窗口大小至关重要
- 微调和编辑方法不适合频繁更新的长上下文场景

**效率视角**：
- 完整交互方法推理成本最高（上下文长度线性增长）
- 检索方法需额外的检索时间和存储成本
- 近期信息方法维持恒定可控的推理成本
- 参数方法避免额外 prompt 开销，但可能需要额外参数选择时间

### 5.5 进一步讨论

1. **不同内容层级**：评估任务可能对应不同抽象层级的记忆（低层：直接提取 vs 高层：需要反思和抽象）
2. **离线评估偏差**：生成数据的记忆机制与被测试的记忆机制之间可能存在分布偏移
3. **综合指标**：需考虑记忆量变化对性能的动态影响

## 六、应用场景

### 6.1 角色扮演与社会模拟

- **核心需求**：记忆需与角色特征一致，用于区分不同角色
- **代表工作**：Character-LLM（经验上传+SFT）、ChatHaruhi（角色记忆提取）、Generative Agents（模拟人类日常行为）、S3（社交网络模拟）
- **设计洞察**：记忆应与角色特征一致；社会模拟需关注多 Agent 间的记忆协调

### 6.2 个人助理

- **核心需求**：记住用户的事实信息和个人风格
- **代表工作**：MemoryBank、RET-LLM、MemoChat、MemGPT、AutoGen、MobileGPT
- **设计洞察**：多采用检索方法；需同时记忆事实信息和个性化风格

### 6.3 开放世界游戏

- **核心需求**：存储过去交互的观察和成功经验
- **代表工作**：Voyager（Minecraft 技能记忆）、GITM、JARVIS-1（多模态记忆）、LARP
- **设计洞察**：关键是反思过去交互、提取可复用经验；外部知识也很关键

### 6.4 代码生成

- **核心需求**：提供开发相关知识，保持代码生成的连续性和一致性
- **代表工作**：ChatDev、MetaGPT、CodeAgent、CodeCoR、RepairAgent
- **设计洞察**：记忆帮助理解开发需求，支持迭代优化

### 6.5 推荐系统

- **核心需求**：保留个性化信息和用户偏好
- **代表工作**：RecAgent、InteRecAgent、RecMind、AgentCF
- **设计洞察**：关键挑战在于如何将个性化信息与 LLM 对齐

### 6.6 特定领域专家系统

| 领域 | 代表工作 | 特点 |
|------|----------|------|
| 医疗 | Huatuo、DoctorGLM、EHRAgent、Agent Hospital | 注入医学知识图谱、领域微调 |
| 金融 | InvestLM、TradingGPT、FinMem、QuantAgent | 存储历史决策、市场信息 |
| 科学 | Chemist-X、ChemDFM、MatChat | 分子数据库、在线文献作为外部知识 |

## 七、局限性与未来方向

### 7.1 参数记忆的更多探索

**现状**：当前以文本记忆为主，参数记忆研究不足。

**挑战**：
- 如何高效将文本信息转化为参数或参数修改
- 参数记忆缺乏可解释性，在高信任度领域（如医疗）是障碍

**潜在方向**：
- 元学习方法（如 MEND）
- 增强参数记忆的可信度和可解释性

### 7.2 多 Agent 系统中的记忆

**关键问题**：
- **记忆同步**：多 Agent 协作中需要统一的知识库
- **Agent 间通信**：依赖记忆维护上下文和解释消息
- **信息不对称**：竞争场景中的关键问题
- **联邦设置**：平衡信息共享（增强协作）vs 隐私保护（保护敏感数据）

### 7.3 基于记忆的终身学习

**核心价值**：Agent 需要像人类一样持续与环境交互、观察和学习。

**挑战**：
- 时间性：记忆需要捕获时间信息，可能导致记忆重叠
- 存储量：终身学习需要存储大量记忆，需要遗忘机制

### 7.4 人形 Agent 的记忆

**核心要求**：
- 记忆需符合人类认知过程（如记忆扭曲、遗忘）
- Agent 应具有知识边界（如扮演儿童的 Agent 不应理解高等数学）

### 7.5 结构增强的记忆

**现状问题**：
- 线性文本记忆库无法感知记忆间的隐含关系
- 无法揭示推理路径

**进展**：
- 图结构（建模事实关系）：AriGraph、MindMap
- 树结构（层次化组织）：MemTree
- 代码结构（过程逻辑）：Voyager

**未来方向**：
- 持续更新和优化记忆结构（而非一次性重构）
- 动态选择最合适的记忆结构
- 从记忆结构中归纳通用规则和模式，增强泛化能力

## 八、论文总结与评价

### 8.1 主要贡献

1. **首次系统综述** Agent 记忆机制，填补了该领域的空白
2. **形式化定义**了记忆模块，明确了狭义和广义的记忆概念
3. **清晰分类体系**：从来源（3类）、形式（2大类6小类）、操作（3类）三个维度系统梳理
4. **全面覆盖** 30+ 代表性模型和 200+ 参考文献
5. **应用导向**：展示了记忆在 7 大类应用场景中的关键作用

### 8.2 关键洞察

| 洞察 | 详情 |
|------|------|
| 记忆 ≠ 简单的 prompt 拼接 | 直接拼接所有历史信息会导致计算成本高、超出上下文限制、位置偏差等问题 |
| 文本与参数互补 | 文本记忆适合短期、可解释场景；参数记忆适合大规模、静态知识 |
| 记忆管理至关重要 | 合并、反思、遗忘三种操作使记忆更加有效 |
| 评估仍需改进 | 离线评估存在偏差，需发展在线评估和动态评估指标 |
| 结构化记忆是趋势 | 图/树/代码等结构化记忆能揭示隐含关系，但仍在初级阶段 |

### 8.3 个人评价

**优点**：
- 框架清晰、体系完整，是入门 Agent 记忆机制的优秀参考资料
- 形式化表达统一，便于理解不同方法之间的关系
- 涵盖了大量代表性工作和详细对比表格

**可改进之处**：
- 论文主要聚焦于文本模态，对多模态记忆的讨论较少
- 对记忆机制的安全性（如记忆注入攻击）讨论不够深入
- 缺少对记忆机制在工业界实际部署中的经验教训分析
